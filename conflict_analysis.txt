# Conflict Analysis & Resolution Strategy

## Root Cause of Conflicts
When attempting to merge the feature branches into `main`, Git throws a `fatal: refusing to merge unrelated histories` error. Furthermore, checking for a common ancestor using `git merge-base main origin/<branch>` returns no common ancestor.

This indicates that the commit history of the `main` branch was rewritten (e.g., via a hard reset and force push, or deleting and recreating the repository/branch). The feature branches are tracking an old history that no longer shares a base with the current `main`.

Because the histories are unrelated, Git treats every file that exists in both branches as a potential conflict if they are not identical. This causes massive conflicts in standard files like `package.json`, `package-lock.json`, configuration files, and common source files.

## Resolution Strategy

Because the branches lack a common ancestor with `main`, a standard git merge or git rebase origin/main will cause massive conflicts across identical files due to the unrelated histories.

The cleanest and safest strategy is **Cherry-picking**:
Instead of trying to force a merge of unrelated histories, we create a fresh branch from the *current* `main` and cherry-pick the specific commits that make up the feature from the old branch. This completely bypasses the unrelated history issue, ensuring we only apply the actual diffs of the feature.

## Implementation Plan

For each branch, follow these steps:

1. **Identify the feature commits:**
   Find the commits introduced in the feature branch. (Typically 1-3 commits).
   `git log --oneline origin/<branch-name>`

2. **Create a fresh branch from current `main`:**
   ```bash
   git checkout main
   git checkout -b <branch-name>-resolved
   ```

3. **Cherry-pick the feature commits:**
   Cherry-pick the commits from oldest to newest.
   ```bash
   git cherry-pick <commit-hash-1> <commit-hash-2> ...
   ```

4. **Resolve specific feature conflicts (if any):**
   When cherry-picking, conflicts will *only* occur if the current `main` has modified the same lines that the feature commit is changing.
   - For `package-lock.json` conflicts: Usually, it's best to resolve `package.json` first, then run `npm install --legacy-peer-deps` to regenerate the lock file cleanly rather than manually merging it.
   - For UI/Logic conflicts (e.g., `src/app/layout.tsx` or `src/components/chat/StreamingTextBlock.tsx`): Manually inspect the conflict markers and keep the new feature additions while preserving changes from the current `main`.

5. **Verify and upload:**
   Run tests and linting.
   ```bash
   npm run build
   npx playwright test
   ```
   If successful, upload the new branch to the remote repository.
